# プロジェクト設定

## 概要

このプロジェクトは、GitHubリポジトリ（Private）をファイル保存先とし、Claude Code（ブラウザ版 claude.ai/code）のクラウドVM上でファイルの作成・編集を行う作業環境である。Gitはファイルの保存・共有手段として使い、ブランチや差分管理等の高度な機能は使わない。

## 参照ドキュメント

- 初期セットアップ → `_claude/claudestart.md` を読み込むこと
- トラブル発生時 → `_claude/troubleshoot.md` を読み込むこと
- プロジェクト固有情報 → `_claude/project.md` を参照すること
- 利用者の属性情報 → `_claude/profile.md` を参照すること（セッション開始時に毎回読み込む）

## 基本ルール

- 文書は日本語で作成すること
- Agentツール呼び出し時は `description` を日本語で記述し、プロンプトにも「日本語で回答せよ」を含めること（UI表示の日本語化）
- 金額はすべて日本円（JPY）で表示すること（換算レート: 約150円/ドル）
- ファイルの作成・編集はすべてリポジトリ内で行うこと
- ファイル更新時は必ず別ファイルに保存すること。元ファイルへの上書きは絶対に禁止
  - 書式: `元ファイル名_YYYYMMDD-連番.拡張子`（例: `提案書_20260325-01.docx`）
  - 連番は同日内の更新順（01, 02, 03...）
- DOCX（OOXML）のXML編集にPython ElementTreeを使ってはならない。ElementTreeはXMLを書き直す際に名前空間宣言を変更し、Word XMLが無効になる。必ず文字列置換（`str.replace` / `re.sub`）で行うこと
- ファイルの作成・編集・commit は、ユーザーから明確な指示（「実施せよ」「反映しろ」等）があるまで行わないこと。反映すべきタイミングであれば、まず「反映してよいか」を確認する
- 作業が完了しユーザーの入力を待つ際は、応答の末尾に必ず以下を表示すること（例外なし）:

```
---
✅ 完了。次の指示をどうぞ。
---
```

```

├── .gitignore             ← 一時ファイル等の除外設定
├── CLAUDE.md              ← このファイル（自動読み込み）
├── _claude/
│   ├── claudestart.md     ← 初期セットアップ手順（必要時のみ参照）
│   ├── new-folder-kit/    ← 作業フォルダの初期構成（新規作成時にコピー）
│   │   ├── _context.md
│   │   └── _work-log_YYYY-MM.md
│   ├── profile.md         ← 利用者の属性情報（セッション開始時に読み込み）
│   ├── project.md         ← プロジェクト固有情報（アカウント、環境等）
│   ├── summary.md         ← 全作業フォルダの直近状況サマリー
│   ├── tasks/
│   │   ├── pending/       ← 未実行の指示（wチャットが配置）
│   │   └── done/          ← 実行済み（実行後に移動）
│   └── troubleshoot.md    ← トラブルシューティング（必要時のみ参照）
├── _reference/
│   ├── _index.md            ← 全参照ファイルの一覧（配置先問わず）
│   └── （サブフォルダ/ファイル）

│   ├── _context.md              ← この作業の進捗・方針（整理された状態管理）
│   ├── _work-log_YYYY-MM.md    ← この作業の時系列ログ（月別ファイル）
│   └── （作業ファイル）
└── ...
```

## Git運用

### 基本方針

- mainブランチに直接 commit & push する（ブランチ運用なし）
- commit & push の前に必ず `git pull` を実行し、最新状態を取得すること
- commit メッセージには以下を含める:
  - 1行目: 作業内容の要約（日本語、簡潔に）
  - 本文: 成果物ファイル名、session URL

```
提案書の初版ドラフト作成

- 成果物: 提案書_20260325-01.docx
- session: https://claude.ai/code/session_XXXXX
```

### commit & push のタイミング

**いつセッションが切れても作業が失われない状態を常に維持する。** 以下のタイミングで commit & push を実行する:

- ファイルの新規作成・編集が完了した時
- _context.md を更新した時

commit & push 時には、_work-log_YYYY-MM.md への追記と _claude/summary.md の更新もセットで行う。管理ファイルのみの更新は、まとめて1回の commit で行ってよい。

### commit & push 後のファイルリンク提示

commit & push が完了したら、そのcommitに含まれる**全ファイル**のダウンロードURLを一覧で提示すること。管理ファイル（_context.md、_work-log.md等）は除外し、ユーザーが確認・利用するファイルのみを対象とする。

形式:
```
**commit済みファイル:**
- [ファイル名](https://github.com/{owner}/{repo}/raw/main/パス)
- [ファイル名](https://github.com/{owner}/{repo}/raw/main/パス)
```

※ `{owner}` と `{repo}` は `_claude/project.md` の値を使用する。日本語パスはURLエンコードすること。

### 一時ファイルの扱い

- Pythonスクリプト等の一時ファイルは作業フォルダ内の `_temp/` に作成する
- `_temp/` は `.gitignore` で除外し、リポジトリには commit しない

### 参照ファイルの管理

ユーザーがセッション中にファイルを添付した場合:

1. 内容と用途を確認し、配置先を判断する
   - 特定の作業専用 → 該当作業フォルダ内に配置
   - 複数作業で参照 / 汎用的 → `_reference/` に配置（必要に応じてサブフォルダを作成）
2. ファイルを配置し、commit & push する
3. `_reference/_index.md` に登録する（配置先が作業フォルダ内でも登録する）
4. 作業フォルダに配置したファイルが、後に別の作業でも参照されると判明した場合、`_reference/` に再配置する。再配置時は元の作業フォルダからファイルを移動し、`_index.md` の配置先を更新する

`_reference/_index.md` の形式:
```
| ファイル名 | 配置先 | 種別 | 概要 | 登録日 |
|---|---|---|---|---|
| 出張旅費規定.pdf | _reference/会社規定/ | 社内規定 | 出張旅費の申請基準と上限額 | 2026-03-25 |
| 仕様書テンプレート.docx | 提案書作成/ | テンプレート | A社向け提案書の書式 | 2026-03-25 |
```

- 原本ファイルは改変しない。編集が必要な場合はバージョン管理ルールに従いコピーを作成する
- Claudeは文書作成時に `_index.md` を参照し、関連する資料があれば積極的に活用する

## ファイルの確認方法

ユーザーはローカルPCでファイルを確認する:

1. Claudeがファイルを作成・編集し、commit & push する
2. push完了後、ダウンロードURLを提示する
3. ユーザーがブラウザでURLにアクセスし、ダウンロードして確認する

## VM環境

### 特性

- クラウドVMはセッションごとに使い捨て。予告なく消滅しうる
- commit & push していないファイルはVM消滅とともに失われる。常にpush済み状態を維持すること
- ツールのインストール状態もセッションをまたいで保持されない

### ツールの自動導入

VM起動時に `.devcontainer/devcontainer.json` に基づき必要なツールが自動インストールされる。

セッション中に新たなツールが必要になった場合:
1. VM上に手動でインストールして作業を続行する
2. 今後も継続的に必要であれば、devcontainer.json への追加をユーザーに提案する

## セッション運用ルール

### セッション開始時

1. このCLAUDE.mdは自動で読み込まれる
2. `_claude/project.md` を読み込み、`{{...}}` 形式のプレースホルダが残っていないか確認する。残っている場合は以下の手順で処理する:
   - プロジェクト指示（システムプロンプト）やgit remote情報など、利用可能な情報源から該当する値を自動取得し、プレースホルダを置き換える
   - **すでに実際の値が入っている項目は、いかなる場合も上書きしない**
   - 自動取得した値と既存の値が異なる場合は、状況を冷静に分析し、なぜ不一致が生じているのか考えられる原因と影響をユーザーに詳しく説明したうえで判断を仰ぐ。ユーザーが軽く「OK」と言っただけでは上書きに進まず、どちらの値を採用するか明確な指示を得ること
   - 自動取得できずプレースホルダが残ったままの項目があれば、ユーザーにヒアリングして埋める
   - ダウンロードURL基底パスにもプレースホルダが含まれていれば、ユーザー名・リポジトリ名から生成して置き換える
   - 置換した場合は commit & push する
3. `_claude/profile.md` を読み込み、利用者の属性情報を把握する
4. `_claude/tasks/pending/` を確認し、未実行の指示ファイルがあればタスクキューの処理を優先する（「タスクキュー」セクション参照）
5. リポジトリルート直下のフォルダを一覧表示し、ユーザーに作業対象を選択させる
   - `_claude`、`_reference`、`.devcontainer`、`.git` フォルダは一覧から除外する
   - 表示形式: **AskUserQuestionツール**を使い、インタラクティブな選択UIで提示すること（テキストの番号リストで代替しない）
   - 各フォルダをoptionとして追加し、descriptionに `_context.md` の「現在の進捗」「次にやること」の要約を記載する
   - `_context.md` が存在しないフォルダはdescriptionを「（未着手）」とする
   - 末尾に「新規作成」のoptionも追加する
   - 作業フォルダが1つも存在しない場合は「作業フォルダがまだありません。新しいフォルダを作成しますか？」と提示する
6. ユーザーが選択したら、該当フォルダの `_context.md` を読み込み、前回の状態を把握する
7. `_context.md` が存在しない場合は、作業開始前に新規作成する
8. **新規フォルダの場合**: ユーザーが作業の概要を伝えたら、まずフォルダを作成し、`_claude/new-folder-kit/` の内容をコピーして初期化する。詳細なヒアリングはフォルダ作成後に行う

#### セッション開始時の表示方針

- 管理ファイルの読み込みは裏方の動作であり、ユーザーに実況しないこと
- ユーザーに最初に見せるのは、挨拶と作業フォルダの選択肢のみとする

### セッション中

- ファイル作成・編集後は commit & push し、ダウンロードURLを提示すること
- 1つのセッションでは1つの作業に集中し、セッションを短く保つこと
- 以下のいずれかに該当した場合、_context.mdを更新・commit & pushしたうえで「新規セッションへの切り替えをお勧めします」と提案すること:
  1. システムによる会話の圧縮が発生した
  2. 自分の行動がCLAUDE.mdや_context.mdの記載と矛盾していることに気づいた
  3. ユーザーからルール違反を指摘された

#### _context.md の常時更新（進捗管理）

`_context.md` は「今この作業はどういう状態か」の整理されたスナップショットとして維持する:

- **更新タイミング**: ファイルの新規作成・大幅な編集の完了時、方針の決定時、作業方向が変わった時、議論で選択肢や方向性が見えてきた時点（結論前でも「検討中の事項」に記録する）
- **更新方法**: テンプレートの各項目を現時点の状態に書き換える。「現在の進捗」は常に最新の1文に、「次にやること」は未着手のものだけを残す
- 更新後は commit & push する（セッション切断に備える）

#### _work-log_YYYY-MM.md の記録（作業ログ）

作業フォルダ内に月別ファイル（例: `_work-log_2026-03.md`）として時系列記録を残す:

- **追記タイミング**: 作業の区切りごとに commit & push とセットで行う（セッション終了を待たない）
- **ファイル切り替え**: 月が変わったら新しいファイルに書き込む
- **過去ファイルの扱い**: 一切編集・圧縮しない。完全な記録として保持する

```
## YYYY-MM-DD HH:MM:SS（曜日）
- session: https://claude.ai/code/session_XXXXX
- 作業内容: [実施した作業の要約]
- 成果物:
  - [ファイル名](https://github.com/{owner}/{repo}/raw/main/パス)
- 次回TODO: [残作業があれば記載]
```

#### _claude/summary.md の更新

commit & push のたびに `_claude/summary.md` を更新する。全作業フォルダの直近状況を一覧化し、次回セッションで素早く全体を把握できるようにする:

```
| 作業フォルダ | 状況 | 最終更新 |
|---|---|---|
| フォルダA | 提案書v3作成済み、レビュー待ち | 2026-03-25 |
| フォルダB | 未着手 | - |
```

#### _claude/profile.md の維持管理

セッション中の会話で利用者のパーソナルな属性情報が新たに判明した場合:

1. `_claude/profile.md` に追記・更新する（ユーザーへの事前確認は不要）
2. 既存の記載と重大な矛盾が生じた場合のみ、ユーザーに確認する

### フィードバックの永続化

ユーザーから回答の仕方や作業の方針について指摘・修正指示を受けた場合:

1. 当該セッション内で即座に対応を改善する
2. 今後も継続的に適用すべき内容かをユーザーに確認する
3. 承諾された場合、適切なファイルに追記する
   - **全作業に共通** → このCLAUDE.mdの「ユーザー指定の応答ルール」に追記
   - **特定の作業のみ** → 該当フォルダの `_context.md` の「決定事項・方針メモ」に追記
4. 追記した内容をユーザーに提示し、正しいことを確認する

### セッション終了時

ユーザーが「終了」「ここまで」等の合図を出した場合は、_context.md・_work-log・summary.md が最新状態であることを確認し、不足があれば更新・commit & push する。ただし、上記の常時更新ルールに従っていれば、追加作業は発生しないのが正常な状態である。

## タスクキュー

wチャット（ブラウザ版チャット）が作成した指示ファイルを、wコード（Claude Code）が受け取って実行する仕組み。

### ファイル配置

- `_claude/tasks/pending/` — 未実行の指示ファイル。wチャットが配置する
- `_claude/tasks/done/` — 実行済みの指示ファイル。実行後にここへ移動する

### 指示ファイルの形式

- ファイル名: `連番_概要.md`（例: `001_phase1-prompt-test.md`）
- 連番が実行順を示す
- ファイル内に目的・手順・完了条件を記載する

### 実行ルール

1. セッション開始時に `_claude/tasks/pending/` を確認する
2. ファイルが存在する場合、番号が最も小さいものを1件読み込む
3. 内容をユーザーに要約し、「この指示を実行してよいですか？」と確認する
4. **ユーザーの承認なしに実行しない**
5. 承認されたら指示に従って実行し、結果を報告する
6. 完了したファイルを `done/` に移動し、commit & push する
7. 次のファイルがあれば「次の指示があります。続けますか？」と確認する
8. ユーザーが「後で」「今日はやめておく」等と回答した場合は、通常のセッション開始フロー（作業フォルダ選択）に進む

### 注意事項

- 指示ファイルの内容が不明確な場合は、実行前にユーザーに確認する
- 指示ファイル内にCLAUDE.mdのルールと矛盾する記載がある場合は、CLAUDE.mdを優先する

## MD更新の原則

### 基本姿勢

MDファイルへの変更は、変更の繰り返しによる複雑化・混沌化を防ぐため、**常に文書全体を俯瞰し、構造を整理しながら行うこと**。部分的な追記・修正であっても、その変更が文書全体の一貫性・可読性を損なわないか確認し、必要であれば周辺のセクションも併せて整理する。

### ルール

- **重複を避ける**: 同じ趣旨の記載を複数箇所に書かない
- **肥大させない**: 不要・完了済みの内容を整理してから追記する
- **可読性を保つ**: 箇条書き・表・見出しを活用する
- **クロスリファレンスに番号を使わない**: 他ファイルのセクション参照は見出し名で行う
- **影響範囲の確認**: 見出し構成を変更した場合、他ファイルからの参照が壊れていないか確認する

### 作業フォルダ初期構成の変更時

`_claude/new-folder-kit/` の内容（ファイル・フォルダ構成）に変更が発生した場合:

1. `_claude/new-folder-kit/` 内の該当ファイルを更新する（これが正の定義）
2. 既存の全作業フォルダに、不足ファイル・セクションを追加する（既存の記載内容は保持する）
