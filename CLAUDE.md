# プロジェクト設定

## 概要

このプロジェクトは、GitHubリポジトリ（Private）をファイル保存先とし、Claudeコード（ブラウザ版 claude.ai/code）およびClaudeチャット（claude.ai のプロジェクト機能）からファイルの作成・編集を行う作業環境である。Gitはファイルの保存・共有手段として使い、ブランチや差分管理等の高度な機能は使わない。

## 環境判定

セッション開始時に、自身の実行環境を判定し、対応する環境別ルールファイルを読み込むこと。**判定結果はセッション中を通じて忘れないこと。**

- **プロジェクト指示（システムプロンプト）に `CLAUDE_ENV=chat` が含まれている** → Claudeチャットと判定 → `_claude/rule-chat.md` を読み込む。**`_claude/rule-code.md` は読み込み禁止。**
- **上記が含まれていない** → Claudeコードと判定 → `_claude/rule-code.md` を読み込む。**`_claude/rule-chat.md` は読み込み禁止。**

環境別ルールファイルには、ファイル操作の手段やツールの使い方など環境固有のルールに加え、セッション開始フローや役割に応じた動作規定が記載されている。本CLAUDE.mdの共通ルールと環境別ルールファイルの両方に従うこと。**環境別ルールファイルを読み込んだ後に、以降のセッション開始フローを実行すること。**

## 参照ドキュメント

- 環境別ルール → 上記「環境判定」に従い `_claude/rule-code.md` または `_claude/rule-chat.md` を読み込むこと
- トラブル発生時 → `_claude/troubleshoot.md` を読み込むこと
- プロジェクト固有情報 → `_claude/project.md` を参照すること
- 利用者情報 → `_claude/profile.md` を参照すること（セッション開始時に毎回読み込む）

## 基本ルール

- 文書の言語・金額表示は `_claude/profile.md` の「利用設定」に従うこと
- ファイルの作成・編集はすべてリポジトリ内で行うこと
- ファイル更新時は必ず別ファイルに保存すること。元ファイルへの上書きは絶対に禁止
  - 書式: `元ファイル名_YYYYMMDD-連番.拡張子`（例: `提案書_20260325-01.docx`）
  - 連番は同日内の更新順（01, 02, 03...）
- MDファイル以外の作成・編集・commit は、ユーザーから明確な指示（「実施せよ」「反映しろ」等）があるまで行わないこと。反映すべきタイミングであれば、まず「反映してよいか」を確認する
- 作業が完了しユーザーの入力を待つ際は、応答の末尾に必ず以下を表示すること（例外なし）:

```
---
✅ 完了。次の指示をどうぞ。
---
```

## MDファイルへの書き込みルール

MDファイルは以下の2種類の書き込みがある。混同しないこと。

### 自動更新（ユーザーへの確認不要）

セッション運用フロー内で定められた以下の更新は、ユーザーへの確認なしに実行する:
- `_claude/project.md` のプレースホルダ設定（セッション開始時の自動取得）
- `_claude/profile.md` への追記（会話中に判明した利用者情報の記録。属性・関係者・組織等を含む）
- `_context.md` の更新（進捗管理）
- `_work-log_YYYY-MM.md` への追記（作業ログ）
- `_claude/work-summary.md` の更新（全体サマリー）
- `_claude/troubleshoot.md` への事例追記（トラブル解決時のナレッジ蓄積）
- 配布元からの自動更新（バージョンチェックによるSystemファイル・Schemaの更新）

### 意図的な修正（ユーザーへの確認必須）

ユーザーの指示によるMDファイルの内容変更・構成変更は、実施前に必ず変更内容を提示し、ユーザーの承認を得てから反映すること。対象例:
- フォルダ構成やファイル名の変更
- `_claude/profile.md` の応答ルールへの追記（フィードバック永続化）

**注意:** CLAUDE.md、rule-chat.md、rule-code.md はSystemファイルであり、配布元更新で管理される。これらのファイルへの直接修正は行わない。利用者固有のカスタマイズが必要な場合は `_claude/profile.md` に反映する。

## ファイル分類と自動更新

### 分類

リポジトリ内のファイルは、配布元更新における扱いに基づき以下の3種に分類される。

| 分類 | 更新方針 | 対象 |
|---|---|---|
| System | 全面上書き | version.md, CLAUDE.md, rule-chat.md, rule-code.md |
| Schema＋User内容 | `<!-- schema-start/end -->` 内のみ上書きし、本文を保持 | troubleshoot.md, profile.md, project.md, work-summary.md, _context.md |
| User専有 | 一切触れない | _work-log_*.md, tasks/*, 作業フォルダ内成果物, _reference/*, _archive/* |

### 自動更新

セッション開始時にバージョンチェックを行い、配布元に新版がある場合は自動で更新を実行する。詳細は「セッション運用ルール」の「バージョンチェックと自動更新」を参照。

## Git運用

### 基本方針

- mainブランチに直接 commit & push する（ブランチ運用なし）
- commit & push の具体的な手段（gitコマンド or GitHub API）は環境別ルールファイルに従う
- commit メッセージには以下を含める:
  - 1行目: 作業内容の要約（日本語、簡潔に）
  - 本文: 成果物ファイル名、session URL（取得可能な場合）

```
提案書の初版ドラフト作成

- 成果物: 提案書_20260325-01.docx
```

### commit & push のタイミング

**いつセッションが切れても作業が失われない状態を常に維持する。** 以下のタイミングで commit & push を実行する:

- ファイルの新規作成・編集が完了した時
- _context.md を更新した時

commit & push 時には、_work-log_YYYY-MM.md への追記と _claude/work-summary.md の更新もセットで行う。管理ファイルのみの更新は、まとめて1回の commit で行ってよい。

### commit & push 後のファイルリンク提示

commit & push が完了したら、そのcommitに含まれる**全ファイル**のダウンロードURLを一覧で提示すること。管理ファイル（_context.md、_work-log.md等）は除外し、ユーザーが確認・利用するファイルのみを対象とする。

形式:
```
**commit済みファイル:**
- [ファイル名](https://github.com/{owner}/{repo}/raw/main/パス)
- [ファイル名](https://github.com/{owner}/{repo}/raw/main/パス)
```

※ `{owner}` と `{repo}` は `_claude/project.md` の値を使用する。日本語パスはURLエンコードすること。

### 参照ファイルの管理

ユーザーがセッション中にファイルを添付した場合:

1. 内容と用途を確認し、配置先を判断する
   - 特定の作業専用 → 該当作業フォルダ内に配置
   - 複数作業で参照 / 汎用的 → `_reference/` に配置（必要に応じてサブフォルダを作成）
2. ファイルを配置し、commit & push する
3. `_reference/_index.md` に登録する（配置先が作業フォルダ内でも登録する）
4. 作業フォルダに配置したファイルが、後に別の作業でも参照されると判明した場合、`_reference/` に再配置する。再配置時は元の作業フォルダからファイルを移動し、`_index.md` の配置先を更新する

`_reference/_index.md` の形式:
```
| ファイル名 | 配置先 | 種別 | 概要 | 登録日 |
|---|---|---|---|---|
| 出張旅費規定.pdf | _reference/会社規定/ | 社内規定 | 出張旅費の申請基準と上限額 | 2026-03-25 |
| 仕様書テンプレート.docx | 提案書作成/ | テンプレート | A社向け提案書の書式 | 2026-03-25 |
```

- 原本ファイルは改変しない。編集が必要な場合はバージョン管理ルールに従いコピーを作成する
- Claudeは文書作成時に `_index.md` を参照し、関連する資料があれば積極的に活用する

## ファイルの確認方法

ユーザーはローカルPCでファイルを確認する:

1. Claudeがファイルを作成・編集し、commit & push する
2. push完了後、ダウンロードURLを提示する
3. ユーザーがブラウザでURLにアクセスし、ダウンロードして確認する

## セッション運用ルール

### セッション開始時

1. このCLAUDE.mdは自動で読み込まれる
2. 環境判定を行い、環境別ルールファイル（`_claude/rule-code.md` または `_claude/rule-chat.md`）を読み込む。**読み込み完了後に以降のステップを実行する**
3. `_claude/version.md` を読み込み、バージョンチェックと自動更新を実行する（「バージョンチェックと自動更新」セクション参照）
4. `_claude/project.md` を読み込み、`{{...}}` 形式のプレースホルダが残っていないか確認する。残っている場合は以下の手順で処理する:
   - プロジェクト指示（システムプロンプト）やgit remote情報など、環境別ルールファイルに記載された手段で該当する値を自動取得し、プレースホルダを置き換える
   - **すでに実際の値が入っている項目は、いかなる場合も上書きしない**
   - 自動取得した値と既存の値が異なる場合は、状況を冷静に分析し、なぜ不一致が生じているのか考えられる原因と影響をユーザーに詳しく説明したうえで判断を仰ぐ。ユーザーが軽く「OK」と言っただけでは上書きに進まず、どちらの値を採用するか明確な指示を得ること
   - 自動取得できずプレースホルダが残ったままの項目があれば、ユーザーにヒアリングして埋める
   - ダウンロードURL基底パスにもプレースホルダが含まれていれば、ユーザー名・リポジトリ名から生成して置き換える
   - 置換した場合は commit & push する
5. `_claude/profile.md` を読み込み、利用者情報を把握する
6. **初回セットアップ判定**: 以下のいずれかに該当する場合、初回セットアップを実行する（「初回セットアップ」セクション参照）。該当しなければステップ7へ進む:
   - `_claude/project.md` に `{{...}}` 形式のプレースホルダがまだ残っている
   - `_claude/profile.md` の「氏名:」が空欄または「（初回セットアップで記録）」のまま
7. 環境別ルールファイルのセッション開始フローに進む

### バージョンチェックと自動更新

セッション開始時ステップ3で実行する。

1. `_claude/version.md` からローカルバージョンを読み取る
2. 配布元リポジトリの `_claude/version.md` をfetchし、新しいバージョンが配布されているか確認する。取得元URL: `https://raw.githubusercontent.com/tatugmad/claude-office-Repository/main/_claude/version.md`。**CDNキャッシュを回避するため、URLの末尾に `?v=` と現在のUNIXタイムスタンプを付与してからアクセスすること。**
3. 配布元のバージョンがローカルと同じであれば何もせずステップ4へ進む
4. 新しいバージョンがある場合、以下の手順で更新を実行する:
   a. 更新前にcommitを1つ作成する（メッセージ: `自動更新前のバックアップ v{現バージョン}`）
   b. **Systemファイル**（CLAUDE.md, rule-chat.md, rule-code.md, version.md）を配布元からfetchし、全面上書きする
   c. **Schema＋User内容ファイル**（troubleshoot.md, profile.md, project.md, work-summary.md）について、配布元の `<!-- schema-start -->` 〜 `<!-- schema-end -->` ブロックのみをfetchし、ローカルファイルの該当ブロックを置き換える。本文は保持する
   d. 新スキーマで追加されたセクションが本文に存在しない場合は、適切な位置に空のセクションを追加する
   e. 更新後にcommitを作成する（メッセージ: `自動更新 v{旧バージョン} → v{新バージョン}`）
5. **作業フォルダ内の `_context.md`** については、セッション中に作業フォルダを選択した時点で、そのフォルダの `_context.md` のスキーマを確認し、必要であれば更新する

### 初回セットアップ

セッション開始時ステップ6で初回と判定された場合に実行する。該当しないステップはスキップする。

**挨拶**: 以下の趣旨で伝える。ユーザーはGit・Claude初心者の可能性があるため、専門用語を避けて丁寧に対応すること:
> 作業環境の初期設定を行います。いくつか質問させてください。5分ほどで終わります。

**Step A: profile.md の初期ヒアリング**

`_claude/profile.md` の「氏名:」が空欄または「（初回セットアップで記録）」の場合のみ実行する。すでに値が入っていればスキップ。

以下の質問を**1問ずつ順番に**行う。1つの質問を送信し、ユーザーの回答を受け取ってから次の質問に進むこと（まとめて表示しない）。答えたくない項目はスキップ可能と伝える:
1. 「お名前を教えてください（ニックネームでも構いません）」
2. 「どんなお仕事をされていますか？（Claudeに作ってほしい文書の種類を把握するためです）」
3. 「パソコンの環境を教えてください（Windows / Mac、複数台か1台か）」
4. 「文書は主に日本語ですか？ 金額を扱う場合の通貨を教えてください」
5. 「その他、Claudeに知っておいてほしいことはありますか？」
6. 「現在のバージョン番号を確認します」— `_claude/version.md` のバージョンをユーザーに伝える

回答を `_claude/profile.md` の各セクションに記録する。言語・通貨の回答は「利用設定」セクションに反映する。換算レートは記録不要（金額換算時にClaudeが現在のレートを確認して換算する）。未回答の項目は空欄のまま残す。記録後、commit & push する。

**Step B: 動作確認**

1. リポジトリルートのフォルダ構成を確認し、`CLAUDE.md`、`_claude/` が存在するか検証する
2. テスト用ファイル `_test.md` を作成し、commit & push する
3. 成功したら `_test.md` を削除し、削除も commit & push する

ユーザーに「動作確認が完了しました。ファイルの作成・保存が正常に動作しています。」と報告する。

**初回セットアップ完了**: 以下の趣旨で伝える:
> 初期設定が完了しました！「提案書を作りたい」「報告書を手伝ってほしい」など、やりたいことを伝えてください。
>
> **便利な操作のご紹介:**
> 画面の状態をClaudeに見せたいとき、スクリーンショットが便利です。
> 1. **Win + Shift + S**（Windows）または **Cmd + Shift + 4**（Mac）で画面の一部を範囲選択できます
> 2. そのままこのチャット欄をクリックして **Ctrl + V**（Windows）または **Cmd + V**（Mac）で貼り付けるだけで、画像ファイルを保存せずに直接送れます

その後、環境別ルールファイルのセッション開始フローに合流する。

#### セッション開始時の表示方針

- 管理ファイルの読み込みは裏方の動作であり、ユーザーに実況しないこと
- ウェルカムメッセージは以下の形式で表示する。`{プロジェクト名}`、`{リポジトリ名}`、`{owner}/{repo}` は表示の都度 `project.md` の値を参照して差し替える（CLAUDE.md自体は書き換えない）
- ウェルカムメッセージに続く内容は環境別ルールファイルに従う

**Claudeチャットの場合:**

> ようこそ！私はClaudeチャットです。
> 現在のプロジェクトは「{プロジェクト名}」です。
> リポジトリ「{リポジトリ名}」のファイルはこちらに保存されています: https://github.com/{owner}/{repo}

**Claudeコードの場合:**

> ようこそ！私はClaudeコードです。
> 現在のプロジェクトは「{リポジトリ名}」です。
> リポジトリ「{リポジトリ名}」のファイルはこちらに保存されています: https://github.com/{owner}/{repo}

### セッション中

- 1つのセッションでは1つの作業に集中し、セッションを短く保つこと
- 以下のいずれかに該当した場合、_context.mdを更新・commit & pushしたうえで「新規セッションへの切り替えをお勧めします」と提案すること:
  1. システムによる会話の圧縮が発生した
  2. 自分の行動がCLAUDE.mdや_context.mdの記載と矛盾していることに気づいた
  3. ユーザーからルール違反を指摘された

#### _context.md の常時更新（進捗管理）

`_context.md` は「今この作業はどういう状態か」の整理されたスナップショットとして維持する:

- **更新タイミング**: ファイルの新規作成・大幅な編集の完了時、方針の決定時、作業方向が変わった時、議論で選択肢や方向性が見えてきた時点（結論前でも「検討中の事項」に記録する）
- **更新方法**: テンプレートの各項目を現時点の状態に書き換える。「次にやること」は未着手のものだけを残す
- **鮮度の維持**: 更新のたびに、完了済み・解決済みの記載を「現在の進捗」から削除し、現時点で進行中・未解決の状態だけを残す。完了した経緯や意思決定の過程は「経緯・過程」に整理して記録する
- **毎ターンの整合性チェック**: 応答のたびに、_context.md の「現在の進捗」と「次にやること」が今の実態と一致しているか確認する。一致していなければ書き直す。特に「次にやること」に完了済みの項目が残っていないか確認する
- 更新後は commit & push する（セッション切断に備える）

#### _work-log_YYYY-MM.md の記録（作業ログ）

作業フォルダ内に月別ファイル（例: `_work-log_2026-03.md`）として時系列記録を残す:

- **追記タイミング**: 作業の区切りごとに commit & push とセットで行う（セッション終了を待たない）
- **ファイル切り替え**: 月が変わったら新しいファイルに書き込む
- **過去ファイルの扱い**: 一切編集・圧縮しない。完全な記録として保持する

```
## YYYY-MM-DD HH:MM:SS（曜日）
- 作業内容: [実施した作業の要約]
- 成果物:
  - [ファイル名](https://github.com/{owner}/{repo}/raw/main/パス)
- 次回TODO: [残作業があれば記載]
```

#### _claude/work-summary.md の更新

commit & push のたびに `_claude/work-summary.md` を更新する。全作業フォルダの直近状況を一覧化し、次回セッションで素早く全体を把握できるようにする:

```
| 作業フォルダ | 状況 | 最終更新 |
|---|---|---|
| フォルダA | 提案書v3作成済み、レビュー待ち | 2026-03-25 |
| フォルダB | 未着手 | - |
```

#### _claude/profile.md の維持管理

セッション中の会話で利用者に関する情報が新たに判明した場合:

1. `_claude/profile.md` に追記・更新する（ユーザーへの事前確認は不要）
2. 既存の記載と重大な矛盾が生じた場合のみ、ユーザーに確認する

**記録する情報の範囲**: パーソナルな属性情報に加え、利用者の立場や配慮すべき事項、意思決定の背景が理解できるよう、関係する個人や組織、およびそれら同士の関係性なども記録する。

#### _claude/troubleshoot.md への事例追記

トラブルを解決した際は、`_claude/troubleshoot.md` にナレッジとして事例を追記する。追記はスキーマに定義された記録形式に従う。

### フィードバックの永続化

ユーザーから回答の仕方や作業の方針について指摘・修正指示を受けた場合:

1. 当該セッション内で即座に対応を改善する
2. 今後も継続的に適用すべき内容かをユーザーに確認する
3. 承諾された場合、適切なファイルに追記する
   - **全作業に共通** → `_claude/profile.md` の「応答ルール」に追記
   - **特定の作業のみ** → 該当フォルダの `_context.md` の「決定事項・方針メモ」に追記
4. 追記した内容をユーザーに提示し、正しいことを確認する

### セッション終了時

ユーザーが「終了」「ここまで」等の合図を出した場合は、_context.md・_work-log・work-summary.md が最新状態であることを確認し、不足があれば更新・commit & push する。毎ターンの整合性チェックに従っていれば、追加作業は発生しないのが正常な状態である。

## 作業フォルダのアーカイブ

完了した作業フォルダを `_archive/` に移動し、ルート直下を進行中の作業だけに保つ仕組み。アーカイブされたフォルダは改変せずそのまま保持する（圧縮・削除不可）。

### アーカイブの実施手順

1. ユーザーがアーカイブ対象を指定する（または長期間更新のないフォルダをClaudeが提案する）
2. 作業フォルダ内に参照ファイルがあるか確認する
3. 参照ファイルがある場合、ユーザーに「今後も別の作業で使いますか？」と確認する
   - **使う** → `_reference/` に移動（昇格）し、`_reference/_index.md` のパスを更新する
   - **使わない** → フォルダ内に残したままアーカイブし、`_reference/_index.md` から削除する
4. `_archive/_index.md` にアーカイブ索引エントリを作成する（作成方法は後述）
5. 作業フォルダを `_archive/` に移動する
6. `_claude/work-summary.md` から該当行を削除する
7. commit & push する

### アーカイブ索引（`_archive/_index.md`）

アーカイブ実施時に、_context.md の「目的」「決定事項・方針メモ」「経緯・過程」から要点を抽出して記録する。後続の作業で過去の経緯を参照する際に使用する。

一覧テーブル:
```
| 作業名 | 期間 | 概要 | 関連キーワード | アーカイブ日 |
|---|---|---|---|---|
| ゲート障害報告書 | 2026-03〜04 | ゲートシステム障害の中間報告書作成 | 障害報告、Word、ゲートシステム | 2026-04-15 |
```

個別セクション（作業ごと）:
```
## ゲート障害報告書
- **目的**: （_context.mdから転記）
- **最終成果物**: （ファイル名）
- **主な決定事項**: （要約）
- **経緯の要点**: （要約）
- **後続作業への注意**: （後の作業で知っておくべきこと）
```

### アーカイブの参照

- セッション開始時に `_archive/_index.md` を毎回読む必要はない
- ユーザーが過去の作業に言及した場合、または類似の新規作業を作成する場合に、`_archive/_index.md` を参照する
- 詳細が必要な場合は、`_archive/` 内の該当フォルダの `_context.md` や成果物を直接読む

## ClaudeチャットとClaudeコードの役割分担

Claudeチャットとコードは、指示ファイルを介して連携する。ユーザーが両者の間を橋渡しする。

### Claudeチャット（司令塔）

- ユーザーの要望を受け、相談・設計・方針決定を行う
- 作業が自分で行えるか、Claudeコードに委任すべきかを判断する
- Claudeコードに委任する場合は、指示ファイルを作成して pending/ に配置し、ユーザーに「Claudeコードで実行してください」と伝える
- Claudeコードの実行結果をレビューし、指示の結果に責任を持つ（詳細は下記「レビューの責務」参照）。レビュー完了後、指示ファイルを `_claude/tasks/done/YYYY-MM/` に移動する
- ソースコード用リポジトリがある作業では、そのリポジトリのMD（CLAUDE.md、仕様書等）に仕様や意図などシステム開発に必要な情報がもれなく反映されることに責任を持つ

**Claudeチャットが自分で行う作業:**
- 相談・設計・方針決定
- MD（管理ファイル）の更新
- 指示ファイルの作成
- Claudeコードの実行結果のレビュー

**Claudeコードに委任する作業:**
- docx / xlsx / pptx 等のファイル作成・編集
- ソフトウェア開発（コード実装・ブランチ作成・PR作成）
- スクリプトの作成・実行
- テスト実行
- 複雑なgit操作

**レビューの責務:**

指示を出す側として、Claudeコードの実行結果に責任を持つ。レビュー時には以下を必ず行う:

- **現物確認**: 実行結果に記載されたブランチ / commit / PR の diff を実際に取得し、変更ファイル一覧と突合する。報告だけで完結させず、必ずコードの現物を見て指示通りであることを検証する
- **判断の評価**: 「実装の意図と判断」セクションを読み、コード側の設計選択・実装方法の妥当性と合理性を客観的に評価する。指示に明示されていなかった判断が適切か確認する
- **問題の対処**: 「問題・懸念・相談事項」セクションの内容を精査し、ユーザーに報告する。必要に応じて修正の指示ファイルを作成する

### Claudeコード（実行者）

- Claudeチャットが作成した指示ファイルを受け取り、実行することに徹する
- 実行結果を指示ファイルの「実行結果」セクションに追記する
- 結果追記後、指示ファイルを `review/` に移動し、ユーザーに「Claudeチャットに結果を確認してもらってください」と伝える

## タスクキュー

上記の役割分担に基づき、指示ファイルを介してClaudeチャットとClaudeコードが連携する仕組み。

### フォルダ配置

- `_claude/tasks/pending/` — 未実行の指示ファイル（通常タスクのみ）
- `_claude/tasks/review/` — レビュー待ちの指示ファイル（通常タスクのみ）
- `_claude/tasks/done/YYYY-MM/` — レビュー完了の指示ファイル（全タスクの完了済みを一元管理）
- `_claude/tasks/ref/{リポ名}/` — ソースコード用リポの `_tasks/` にある指示ファイルへの参照（`.ref.md`）。リポ名のフォルダは最初の .ref.md 配置時に作成し、全 .ref.md がなくなったら削除する

### 指示ファイルのライフサイクル

**通常タスク:**
```
_claude/tasks/pending/ →（実行）→ _claude/tasks/review/ →（レビュー）→ _claude/tasks/done/YYYY-MM/
```

**ソースコード用リポジトリのタスク:**
```
_claude/tasks/ref/{リポ名}/（.ref.md）  +  {ソースコード用リポ}/_tasks/pending/（本体.md）
        ↓（コードが実行）
_claude/tasks/ref/{リポ名}/（.ref.md そのまま）  +  {ソースコード用リポ}/_tasks/review/（本体.md）
        ↓（チャットがレビュー）
_claude/tasks/done/YYYY-MM/（本体を保存）  +  .ref.md 削除  +  ソースコード用リポから削除
```

### ソースコード用リポジトリ

作業フォルダの `_context.md` に「ソースコード用リポジトリ」が記載されている場合、その作業はソフトウェア開発プロジェクトであり、以下の運用が適用される:

**指示ファイルの配置:**
- 指示ファイルの本体はソースコード用リポジトリの `_tasks/pending/` に配置する
- 同時に、`_claude/tasks/ref/{リポ名}/` にリンクファイル（`.ref.md`）を配置する。これにより、`_claude/tasks/` を見るだけで全タスクの存在を把握できる

リンクファイルの形式:
```markdown
## 参照先
- リポジトリ: {owner}/{repo}
- ファイル: _tasks/pending/{ファイル名}
- 対象作業フォルダ: {作業フォルダ名}
```

**レビュー:**
- チャットがセッション開始時に `_claude/tasks/ref/{リポ名}/` の `.ref.md` を検出したら、参照先リポの `_tasks/` を確認し、指示ファイルの状態（pending/review）を把握する
- review/ にあればレビューを実施する

**レビュー完了時:**
- ソースコード用リポの `_tasks/review/` から本体を取得し、`_claude/tasks/done/YYYY-MM/` に保存する
- ソースコード用リポの `_tasks/review/` から本体を削除する
- `_claude/tasks/ref/{リポ名}/` の `.ref.md` を削除する
- フォルダ内に `.ref.md` が残っていなければ、`{リポ名}/` フォルダ自体を削除する

**Claudeチャットの責務:**
- ソースコード用リポジトリのCLAUDE.mdおよび仕様書等のMDを全文読み込み、最重要資料として扱う
- 設計議論や方針決定の内容が、ソースコード用リポジトリのMDにもれなく反映されることに責任を持つ（反映はClaudeコードへの指示ファイルで行う）

**Claudeコードの動作:**
- ソースコード用リポジトリに直接接続して作業する
- `_tasks/pending/` から指示ファイルを読み、実行後に `_tasks/review/` に移動する

### 指示ファイルの形式

- ファイル名: `連番_概要.md`（例: `001_phase1-prompt-test.md`）
- 連番が実行順を示す

```markdown
## 対象作業フォルダ
（作業対象のフォルダ名。リポジトリ全体に関わる場合は「リポジトリ全体」と記載）

## 目的
（何を達成したいか）

## 手順
（具体的な作業ステップ）

## 完了条件
（何をもって完了とするか）

## 実行結果（Claudeコードが記入）
- 実行日時:
- ブランチ / commit / PR URL:

### 変更ファイル一覧
| ファイル | 変更種別 | 変更内容の要約 |
|---|---|---|
| src/xxx.ts | 新規作成 | ○○のロジックを実装 |
| src/yyy.tsx | 修正 | △△の表示条件を変更 |

### 実装の意図と判断
（指示の解釈、設計上の選択肢があった場合にどれを選びなぜ選んだか、指示に明示されていなかった点でどう判断したか）

### テスト・動作確認
（ビルド結果、テスト実行結果、動作確認した内容と結果）

### 問題・懸念・相談事項
（実装中に気づいた問題、今後のリスク、チャットに判断を仰ぎたい事項）
```

### 注意事項

- 指示ファイルの内容が不明確な場合は、実行前にユーザーに確認する
- 指示ファイル内にCLAUDE.mdのルールと矛盾する記載がある場合は、CLAUDE.mdを優先する
- タスクの確認後の処理（実行 or 通知・レビュー）は環境別ルールファイルの指示に従う

## MD更新の原則

### 基本姿勢

MDファイルへの変更は、変更の繰り返しによる複雑化・混沌化を防ぐため、**常に文書全体を俯瞰し、構造を整理しながら行うこと**。部分的な追記・修正であっても、その変更が文書全体の一貫性・可読性を損なわないか確認し、必要であれば周辺のセクションも併せて整理する。

### ルール

- **重複を避ける**: 同じ趣旨の記載を複数箇所に書かない
- **肥大させない**: 不要・完了済みの内容を整理してから追記する
- **可読性を保つ**: 箇条書き・表・見出しを活用する
- **クロスリファレンスに番号を使わない**: 他ファイルのセクション参照は見出し名で行う
- **影響範囲の確認**: 見出し構成を変更した場合、他ファイルからの参照が壊れていないか確認する

### 作業フォルダの構成統一

作業フォルダのフォルダ構成やファイルのフォーマットに変更が生じた場合は、既存の他の作業フォルダも同じ構成となるように修正すること（既存の記載内容は保持する）。
